## Task Analysis
- Main objective: Add an adapter-only abstraction for router, gesture, toast, device info, keep-awake, storage, and haptics in a new `utils/platform-adapters.js`, plus mocks, tests, and migration docs, without migrating current pages/components off direct `hm*` usage yet.
- Identified dependencies: Tasks `73` and `74` already locked the Zepp `3.6+` mainline and v3 manifest direction; current raw platform calls live in `app.js`, `page/game.js`, `page/index.js`, `page/setup.js`, `page/settings.js`, `page/history.js`, `page/history-detail.js`, and `page/summary.js`; reusable fallback patterns already exist in `utils/storage.js`, `utils/haptic-feedback-settings.js`, `utils/screen-utils.js`, and `tests/helpers/hmfs-mock.js`; the repo test runner is `node --test`, not Jest, so mocks must be Jest-compatible without adding Jest.
- System impact: Low immediate runtime impact because current consumers stay unchanged, but medium contract impact because later migration tasks will depend on the new export names, fallback semantics, and mock behavior staying stable.

## Chosen Approach
- Proposed solution: Implement one lightweight adapter module with named exports for all seven adapters, backed by lazy capability detection and deterministic no-op or in-memory fallbacks; pair it with one dedicated mock module, one adapter-focused test suite or smoke harness, and one migration document.
- Justification for simplicity: This keeps the abstraction surface in one place, reuses fallback patterns the repo already trusts, avoids premature per-adapter file splitting, and honors the approved scope by not refactoring pages/components yet. Rejected alternatives: splitting seven thin adapters across many files with no current reuse benefit, and immediately converting page/component consumers to adapters, which is out of scope and raises regression risk.
- Components to be modified/created: `utils/platform-adapters.js`, `tests/__mocks__/platform-adapters.js`, `tests/platform-adapters.test.js` (or one equivalently named adapter-focused test file), `docs/PLATFORM_ADAPTERS.md`, and this plan file. Expected consumers to remain unchanged for now: `app.js`, `page/game.js`, `page/index.js`, `page/setup.js`, `page/settings.js`, `page/history.js`, `page/history-detail.js`, and `page/summary.js`.

## Implementation Steps
1. Lock the adapter contract before coding: inventory the current raw platform touchpoints (`hmApp.gotoPage` / `goBack` / `registerGestureEvent`, `hmUI.showToast`, `hmSetting.getDeviceInfo` / `setBrightScreen` / `setBrightScreenCancel`, `hmSensor.createSensor`, and existing storage helpers) and define the exact stable named exports and method signatures that Tasks `76+` will target. Assumption checkpoint: the adapter module is additive only, pages/components stay on raw globals in this task, and no new persistence schema or routing behavior is introduced.
2. Implement `utils/platform-adapters.js` as the single runtime contract: add named exports `router`, `gesture`, `toast`, `deviceInfo`, `keepAwake`, `storage`, and `haptics`; use lazy runtime resolution inside methods so the module can load under Node tests without a Zepp runtime; prefer Zepp `3.6`-compatible behavior where it can be expressed safely, otherwise fall back to current `hm*` globals or deterministic no-op or in-memory behavior. Elevated-risk mitigation: avoid top-level hard imports that would make `node --test` fail when Zepp SDK modules are unavailable.
3. Define adapter internals to mirror existing repo patterns instead of inventing new infrastructure: reuse `Map`-backed fallback storage patterns from `utils/storage.js`, device metric defaults and round-screen heuristics from `utils/screen-utils.js`, current keep-awake semantics from `app.js` and `page/game.js`, and current haptic behavior expectations from `page/game.js` and `page/summary.js`. Correctness checkpoint: every adapter method returns a predictable value or state even when the underlying runtime API is missing.
4. Create `tests/__mocks__/platform-adapters.js` as a manual-import mock layer that is Jest-compatible but runner-agnostic: track router history, redirect or back events, toast visibility and state, isolated storage contents, keep-awake status, configurable device info, recorded haptic calls, and gesture registrations plus trigger helpers; expose a single reset helper that clears all state between tests. Risk mitigation: do not depend on `jest.fn()` so the mocks remain usable in the repo's current `node:test` environment.
5. Add adapter-focused tests using the current Node test stack: verify export shape from `utils/platform-adapters.js`, fallback behavior without Zepp runtime, mock isolation and reset semantics, and a smoke or integration flow that exercises all seven adapters through the mock module without touching existing pages/components. If module cache isolation is needed, reuse `tests/helpers/project-paths.js` and dynamic imports rather than changing production consumers.
6. Write `docs/PLATFORM_ADAPTERS.md` as the migration handoff document: explain why the adapter layer exists, list each adapter API, show before or after examples based on current raw calls in pages and utilities, call out breaking changes and non-goals, and state clearly that Task `75` introduces the contract only while page/component adoption happens in later tasks. Documentation checkpoint: examples align with actual current file paths and do not imply that existing pages have already been refactored.
7. Run validation in increasing scope: `npm test` targeted to the new adapter tests first, then full `npm test`, then `npm run complete-check`. Rollback note: if a runtime-specific adapter behavior cannot be proven safely without bundling later migration work, keep the adapter on a documented graceful fallback or no-op path and record the deferred runtime swap in `docs/PLATFORM_ADAPTERS.md` instead of expanding scope.

## Validation
- Success criteria: `utils/platform-adapters.js` exports all seven adapters with the required method names; adapter methods load safely in Node without Zepp runtime and provide documented fallbacks; `tests/__mocks__/platform-adapters.js` supports router history, toast state, storage isolation, haptics records, gesture triggering, configurable device info, and full reset between tests; smoke tests pass without refactoring existing pages/components; `docs/PLATFORM_ADAPTERS.md` fully documents the pattern, examples, breaking changes, and expected future usage.
- Checkpoints: Pre-implementation checkpoint confirms the exact export contract, current raw API inventory, and no-consumer-refactor scope; implementation checkpoint A verifies the new adapter module loads cleanly under Node and each method has deterministic fallback semantics; implementation checkpoint B verifies mock reset or isolation behavior and smoke-test coverage for all seven adapters; documentation checkpoint verifies every adapter has before or after migration examples tied to current repo files; final checkpoint verifies `npm test` and `npm run complete-check` pass without new runtime errors or export-shape drift.
