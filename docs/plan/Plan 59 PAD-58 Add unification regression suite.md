## Task Analysis
- Main objective: Add a dedicated unification regression suite for the Task 53-58 persistence unification work, with deterministic headless coverage for canonical schema validation, unified active-session storage behavior, migration/runtime consistency, matchStartTime initialization and immutability, and app-restart persistence without GlobalData handoff coupling.
- Identified dependencies: Task 53 (`utils/match-state-schema.js`, `docs/schema/match-session.*`), Task 54 (`utils/active-session-storage.js`, `tests/helpers/hmfs-mock.js`, `tests/fixtures/legacy-sessions/*`), Task 55 (canonical contract fixtures in `tests/fixtures/match-session-examples.js`), Task 56/57/58 runtime + lifecycle behaviors in `app.js`, `page/setup.js`, `page/game.js`, and existing coverage patterns in `tests/active-session-storage.test.js`, `tests/active-session-storage.integration.test.js`, `tests/match-state-schema.test.js`, `tests/match-storage.test.js`, `tests/match-session-init.test.js`, `tests/app-undo-integration.test.js`.
- System impact: Medium on test architecture and CI signal quality, low on production runtime unless a minimal hook is strictly required; key risks are flaky timing assertions, accidental over-coupling to implementation details, and CI trigger ambiguity for `v*` pushes.
- Assumptions Made: Dependencies 53-58 are functionally complete on the target branch; Node test runner (`node --test --test-isolation=process --test-concurrency=1`) remains the canonical harness; current hmFS mock strategy remains valid for headless storage/persistence verification; `v*` refers to push branches (not release tags) unless clarified.
- Targeted Clarifications: 1) Confirm whether `v*` means branches, tags, or both for CI triggers. 2) Confirm whether the team wants a dedicated script (for example `test:unification`) or only inclusion in `npm test`. 3) Confirm acceptable scope for optional hooks: only test bridges/global callbacks (existing project pattern) vs adding any production exports.

## Chosen Approach
- Proposed solution: Create a focused regression suite slice that reuses existing helpers and fixtures, grouped as a dedicated unification suite (unit + integration), add only gap-focused scenarios instead of duplicating broad page tests, wire CI to run the suite headlessly on PR + push (`main` and `v*`), and add test hooks only if existing public page/app methods cannot provide deterministic assertions.
- Justification for simplicity: Deepthink compared three approaches: (A) large end-to-end simulator-style flow suite across all pages (high maintenance, overengineered), (B) targeted headless Node regression suite reusing current hmFS/page import patterns (chosen, smallest effective change), and (C) introducing a new generic test framework abstraction first (unnecessary architecture overhead). Approach B gives high confidence with minimal production churn.
- Components to be modified/created: New suite files under `tests/unification-regression/` (unit + integration split), optional fixtures under `tests/fixtures/unification-regression/`, selective additions to existing storage/schema/session tests where coverage already lives, optional minimal test bridge helper in `tests/helpers/` only if required, CI workflow update in `.github/workflows/ci.yml`, and run instructions update in `README.md` (or `docs/GET_STARTED.md` if preferred by existing docs ownership).
