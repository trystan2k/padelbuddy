## Task Analysis
- Main objective: Replace the current file/legacy persistence stack with a Zepp OS 3.6+ `LocalStorage`-first model for active session, match history, and app settings, while keeping current in-app behavior stable for new users and retaining only forward-looking schema migration support.
- Identified dependencies: Task `75` already introduced `utils/platform-adapters.js` with a generic `storage` adapter that resolves `__zosStorage` / `localStorage` / `LocalStorage`; Task `73` already moved the app to the Zepp OS 3.6+ mainline. Current persistence touchpoints are `utils/match-storage.js`, `utils/active-session-storage.js`, `utils/storage.js`, `utils/match-history-storage.js`, `utils/haptic-feedback-settings.js`, `utils/app-data-clear.js`, `app.js`, and page consumers in `page/index.js`, `page/setup.js`, `page/game/persistence.js`, `page/history.js`, `page/history-detail.js`, `page/game-settings.js`, and `page/summary.js`. Canonical payload rules already live in `utils/match-state-schema.js` and `utils/match-history-types.js`. Test coverage currently leans heavily on `hmFS` mocks in `tests/helpers/hmfs-mock.js` and related storage/session suites.
- System impact: High utility/test impact with moderate app-flow risk. The safest change path is to centralize persistence internals and keep existing public session/history/settings APIs stable for pages, while deleting Zepp OS v1-era file migration logic, startup migration hooks, and `hmFS` cleanup behavior once the LocalStorage path is validated.

## Chosen Approach
- Proposed solution: Introduce a shared `utils/persistence.js` core built on `utils/platform-adapters.js#storage`, then refactor the existing domain-specific persistence modules to use that core instead of `hmFS` or legacy runtime storage. Keep `utils/match-state-schema.js` as the source of truth for session payload migration/normalization, add a lightweight storage-schema bootstrap for future LocalStorage-only transitions, remove startup legacy migration from `app.js`, and delete `@zos/fs`/legacy compatibility code that only exists for the old v1 rollout.
- Justification for simplicity: Deepthink comparison favored this over two more complex alternatives: (A) rewriting every page to call a brand-new generic persistence API directly, which expands the regression surface, and (B) giving each storage module its own LocalStorage implementation, which duplicates error handling and schema/version logic. Reusing Task `75`'s adapter plus the existing session/history/settings facades is the smallest effective path because most page imports stay intact while the storage internals collapse to one modern strategy.
- Components to be modified/created: Create `utils/persistence.js`; refactor `utils/match-storage.js`, `utils/active-session-storage.js`, `utils/match-history-storage.js`, `utils/haptic-feedback-settings.js`, `utils/app-data-clear.js`, and `app.js`; remove or retire `utils/storage.js` after consumers are moved; audit `page/game/persistence.js` and other page consumers for hardcoded legacy schema assumptions; update `docs/schema/match-session.md` and any persistence docs that still describe `hmFS`/legacy migration; replace `hmFS`-centric tests with LocalStorage/platform-adapter-based coverage and retire obsolete migration suites/helpers when no longer needed.

## Implementation Steps
1. Lock the new scope before editing: confirm Task `76` is implemented against the user override rather than the original Taskmaster backward-compatibility wording, inventory the live persistence keys/contracts still required by the app (`ACTIVE_MATCH_SESSION`, `padel-buddy.match-history`, `padel-buddy.haptic-feedback-enabled`), and explicitly mark `padel-buddy.match-state`, legacy active-session files, startup migration, and `hmFS` fallback as removal targets.
2. Create `utils/persistence.js` as the single runtime persistence core: expose simple JSON-safe operations such as `saveState`, `loadState`, `deleteState`, and targeted clear helpers on top of `platform-adapters.storage`; centralize try/catch behavior, validator hooks, and storage-schema marker keys (for example `STORAGE_SCHEMA_VERSION` and optional metadata/backup keys) so all runtime persistence uses one adapter path.
3. Implement the storage-schema framework in the new core, but scope it to LocalStorage-first forward evolution only: add `CURRENT_STORAGE_SCHEMA_VERSION`, an idempotent `ensureStorageSchema()`/`migrateStorageSchema()` pipeline, and migration registry plumbing for future schema bumps; for this task, the initial migration should be a no-op bootstrap/normalization step for fresh or already-current LocalStorage data, not a legacy file-to-LocalStorage conversion. Elevated-risk mitigation: if backup/rollback support complicates the design, limit it to per-key shadow copies used only inside future LocalStorage migrations instead of resurrecting legacy migration infrastructure.
4. Refactor active-session persistence to the new core while preserving app-facing APIs: move `utils/match-storage.js` and `utils/active-session-storage.js` onto `utils/persistence.js` + `utils/match-state-schema.js`, keep `getActiveSession` / `saveActiveSession` / `updateActiveSession` / `clearActiveSession` semantics for existing callers, delete legacy candidate selection and cleanup branches, and remove `migrateLegacySessions` plus related file-path constants once `app.js` and tests no longer depend on them.
5. Refactor history and settings persistence to the same core: replace `hmFS` reads/writes in `utils/match-history-storage.js` and `utils/haptic-feedback-settings.js` with LocalStorage-backed persistence helpers, keep current exported APIs and payload shapes where current pages depend on them, simplify `utils/app-data-clear.js` so it clears only the new storage keys, and delete direct file-overwrite cleanup logic that only exists to work around old `hmFS` behavior.
6. Audit schema assumptions and page references before final cleanup: update any page or utility code still hardcoding legacy schema values or storage-era behavior (notably `page/game/persistence.js`, resume/history flows, and any `schemaVersion: 1` fallbacks), keep `utils/match-state-schema.js` as the canonical serializer/deserializer for session payloads, and ensure page consumers either continue using the same facades or switch in a narrowly scoped way only where a facade no longer makes sense.
7. Remove legacy-only code and test infrastructure after the new path is green: delete `utils/storage.js` if no remaining imports exist, remove the startup migration flag/hook from `app.js`, drop obsolete `hmFS` helpers and legacy migration tests, add LocalStorage/platform-adapter test helpers where needed, and rewrite affected unit/integration tests (`tests/match-storage.test.js`, `tests/match-history-storage.test.js`, `tests/haptic-feedback-settings.test.js`, page-flow suites, and any unification regression suites) to validate the modern storage path instead of v1-era file behavior.
8. Run validation and document the decision: execute targeted persistence/page tests during the refactor, then run `npm run complete-check`; if simulator validation is available, smoke-test start match -> resume match -> finish match -> history -> clear data on at least one square and one round target; update `docs/schema/match-session.md` plus any storage migration docs to state that mainline Zepp 3.6+ persistence is LocalStorage-first with no legacy user-data migration support.

## Validation
- Success criteria: All active-session, history, settings, and clear-data flows persist through the LocalStorage-first adapter path; no production module depends on `hmFS`, legacy runtime storage keys, or startup legacy migration; `utils/match-state-schema.js` still validates and normalizes persisted session payloads correctly; page behavior for resume/history/settings remains unchanged for the Zepp 3.6+ app line; `npm run complete-check` passes.
- Checkpoints: Pre-implementation checkpoint after Step 1 confirms the scope lock and removal list; implementation checkpoint after Steps 2-3 confirms the new persistence core and schema bootstrap are idempotent under tests; implementation checkpoint after Steps 4-6 confirms session/history/settings/page flows work with unchanged public APIs or intentional import updates; cleanup checkpoint after Step 7 confirms no remaining `hmFS`/legacy imports via repository search; final checkpoint after Step 8 confirms QA command success and documentation accurately records the no-legacy-support decision. Rollback/mitigation: if active-session behavior regresses, keep existing public facades and revert only the internal storage swap instead of restoring legacy migration; if LocalStorage runtime gaps appear on Zepp 3.6+ targets, harden `platform-adapters.storage` first and use its in-memory fallback for non-destructive failure handling rather than reintroducing `@zos/fs`.
