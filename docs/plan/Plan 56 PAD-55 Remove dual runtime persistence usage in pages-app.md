## Task Analysis
- Main objective: Refactor app/session lifecycle flows to use only the Unified active-session storage service APIs (`getActiveSession`, `saveActiveSession`, `clearActiveSession`, `migrateLegacySessions`) for session persistence, while preserving current behavior across setup -> game -> summary and restart paths.
- Identified dependencies: Task 53 canonical contract in `utils/match-state-schema.js`; Task 54 unified service in `utils/active-session-storage.js`; current page call sites in `app.js`, `page/game.js`, `page/setup.js`, `page/index.js`, `page/summary.js`; flow helper `utils/start-new-match-flow.js`; test harnesses in `tests/home-screen.test.js`, `tests/setup-flow.test.js`, `tests/game-screen-layout.test.js`, `tests/summary-screen.test.js`, `tests/app-undo-integration.test.js`, and `tests/active-session-storage.integration.test.js`.
- System impact: High on match continuity and restart resilience because persistence is touched in hot gameplay paths (`page/game.js`) and app lifecycle (`app.js`); medium on migration safety and regression surface in page integration tests.
- Assumptions made: (1) Task wording `pages/app` maps to this repository structure `app.js` + `page/*`; (2) Task 56 removes dual persistence read/write paths but does not fully remove globalData handoff channels (explicitly targeted by Task 57); (3) compatibility module `utils/match-storage.js` may remain for non-page callers/tests, but page/app call sites should move to unified service APIs directly.
- Targeted clarifications: 1) Should Task 56 include removing all `globalData.pending*` handoff usage now, or keep it for Task 57? 2) Do you want `utils/match-storage.js` deprecated for all new code immediately, or only page/app call sites in this task? 3) For app hard-kill fallback, if canonical write fails, should we prefer best-effort stale snapshot retention or explicit failure logging with no fallback write?

## Chosen Approach
- Proposed solution: Execute a page/app-first cutover that replaces read/write/clear session operations in `app.js` and `page/*` with direct unified service API calls, removes direct `utils/storage.js` runtime session access from those flows, keeps migration startup behavior in `app.js`, and updates impacted tests to validate the same user-visible behavior.
- Justification for simplicity: Deepthink compared (A) full storage-layer rewrite including immediate handoff elimination, (B) targeted page/app cutover with existing handoff behavior preserved, and (C) introducing a new session repository abstraction; (B) is simplest and safest because it satisfies Task 56 acceptance, minimizes architectural churn, and avoids overlap/merge risk with Task 57.
- Components to be modified/created: Modify `app.js`, `page/game.js`, `page/setup.js`, `page/index.js`, `page/summary.js`, `utils/start-new-match-flow.js` (if needed for restart clear-path consistency), and related tests (`tests/home-screen.test.js`, `tests/setup-flow.test.js`, `tests/game-screen-layout.test.js`, `tests/summary-screen.test.js`, `tests/app-undo-integration.test.js`, `tests/active-session-storage.integration.test.js`, `tests/start-new-match-flow.test.js`); no new production module is required unless runtime->canonical snapshot logic must be extracted for shared use.
