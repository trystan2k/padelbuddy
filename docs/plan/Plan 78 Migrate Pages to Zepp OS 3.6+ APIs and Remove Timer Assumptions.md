## Task Analysis
- Main objective: Migrate the eight page modules to the Task 75-77 Zepp OS 3.6+ seams by replacing direct page-level `hmApp`/`hmUI.showToast`/`hmSensor`/`hmSetting` usage with adapter-backed navigation, toast, haptics, keep-awake, and gesture flows, while removing all standard JS timer assumptions from page logic.
- Identified dependencies: Task `75` already provides `utils/platform-adapters.js` (`router`, `toast`, `haptics`, `keepAwake`, `gesture`, `storage`); Task `76` already moved session/history/settings persistence behind `utils/persistence.js` and its domain facades (`utils/match-storage.js`, `utils/match-history-storage.js`, `utils/haptic-feedback-settings.js`); Task `77` already established family-aware `getScreenMetrics()` for `w390-s`, `w454-r`, `w466-r`, and `w480-r`. Current page hotspots are direct navigation in `page/index.js`, `page/setup.js`, `page/game.js`, `page/summary.js`, `page/history.js`, `page/history-detail.js`, `page/settings.js`, and `page/game-settings.js`; timer usage in `page/game.js`, `page/summary.js`, `page/history-detail.js`, and `page/settings.js`; and adjacent helper seams in `page/game/persistence.js` and `utils/start-new-match-flow.js`.
- System impact: This is a Device App page refactor with no expected `app.json` or permission changes, but it touches every interactive flow: home/setup navigation, active-session persistence, scoring responsiveness, summary haptics, history list/detail actions, and settings confirmation UX. Screen adaptation must continue to flow through the Task `77` family-first metrics contract so square and round targets stay stable.

## Chosen Approach
- Proposed solution: Refactor page modules in place to consume the existing adapter and persistence seams instead of introducing any new scheduler or compatibility layer: swap raw page-level runtime calls to `router`/`toast`/`haptics`/`keepAwake`/`gesture`, keep session/history/settings reads and writes on the existing Task `76` facades unless a page-owned helper still hardcodes storage-era behavior, and delete timer-based UX by converting it to explicit state transitions and immediate persistence.
- Justification for simplicity: Deepthink comparison rejected two heavier options: (A) adding a new native-timer abstraction or page scheduler to recreate `setTimeout` semantics for only a handful of flows, and (B) bypassing the current storage facades by rewriting every page directly against `utils/persistence.js`. The simplest effective path is to reuse the current seams, remove timers instead of emulating them, and keep behavior changes localized to the pages and the small helper modules they already own.
- Components to be modified/created: Modify `page/index.js`, `page/setup.js`, `page/game.js`, `page/summary.js`, `page/history.js`, `page/history-detail.js`, `page/settings.js`, `page/game-settings.js`, and the adjacent helper seams that those pages directly depend on for migrated behavior (`page/game/persistence.js` and `utils/start-new-match-flow.js` if their current raw runtime assumptions block the page migration); update the relevant tests in `tests/home-screen.test.js`, `tests/game-screen-layout.test.js`, `tests/game-persistence.test.js`, `tests/summary-screen.test.js`, `tests/settings-navigation.test.js`, and any timer/static-analysis coverage added for this task.

## Implementation Steps
1. Lock scope and capture the migration baseline before editing: inventory every direct `hmApp`, `hmUI.showToast`, `hmSensor`, `hmSetting`, and timer call that still originates from the eight target pages; confirm `app.json` still targets API `3.6` with the four screen families from Task `77`; and decide which page imports can stay on Task `76` domain facades because they already route through `utils/persistence.js`. Pre-implementation assumption check: `page/game/persistence.js` and `utils/start-new-match-flow.js` are included only if their page-facing contracts still hide raw runtime or storage-era assumptions.
2. Migrate the core page platform calls first (`page/index.js`, `page/setup.js`, `page/game.js`, `page/summary.js`): replace direct page-owned navigation/toast/haptic/keep-awake/gesture calls with the Task `75` adapters, keep `getScreenMetrics()` as the only screen-family seam, and update any helper-bound page navigation path (notably the new-match flow and game-page redirects) so the core flow no longer depends on raw `hmApp` at the page boundary. Correctness checkpoint: the start/resume/setup/game/summary route chain still behaves the same on square and round layouts after the import swap.
3. Remove timer-dependent logic from the core gameplay flow instead of recreating it: convert `page/game.js` manual-finish confirmation from a three-second timeout window to an explicit two-step confirm state cleared by user action, navigation, or page teardown; replace runtime-state persistence debounce with signature-based immediate saves after accepted scoring actions plus lifecycle flushes; and reduce `page/summary.js` load haptics from delayed pulse timers to one immediate adapter-backed trigger or one adapter-level pattern call with no page timers. Risk mitigation: `page/game.js` is the highest-risk file, so keep the current interaction fingerprint and dedupe logic intact while deleting only the timer scheduling itself.
4. Migrate the history and settings pages onto the same seams: replace direct navigation and toast calls in `page/history.js`, `page/history-detail.js`, `page/settings.js`, and `page/game-settings.js` with adapters; keep history/settings persistence on `utils/match-history-storage.js`, `utils/haptic-feedback-settings.js`, `utils/app-data-clear.js`, or a narrowly scoped `utils/persistence.js` helper where appropriate; and normalize detail-page params to the router query payload shape used by `platform-adapters.router`. Early validation checkpoint: confirm history-detail still resolves a `matchId` when navigation switches from legacy `param` payloads to adapter-built query strings.
5. Redesign confirmation and settings behaviors so they are lifecycle- or action-driven, not timer-driven: make history deletion and clear-data confirmation modes persist until the next explicit user action, navigation, or `onDestroy`; remove the delayed post-toast navigation from `page/settings.js` so clear-data returns to home immediately after the clear result is known; and keep `page/game-settings.js` on immediate save semantics so toggles persist as soon as the user changes them. Rollback/mitigation note: if explicit confirm state feels too sticky during simulator testing, clear it on any alternate actionable tap and page teardown rather than reintroducing timers.
6. Refactor page-specific list and render paths only where the migration requires it: keep using the existing family-aware layout resolution from Task `77`, but move history/settings list refreshes toward one 3.6-compatible widget update path (`SCROLL_LIST` creation plus `setProperty(UPDATE_DATA)` where data changes) and avoid any polling or delayed re-render assumptions. Simplicity guard: do not invent a new shared list abstraction unless both history and settings need the exact same extracted logic to stay readable.
7. Update automated coverage and verify the migrated behavior in increasing cost order: add static-analysis coverage for timer removal from the eight page files; update page tests to expect adapter-backed navigation/toast/haptic behavior and explicit confirmation states; then run targeted suites for home/game/summary/settings/history-adjacent flows before the full `npm run complete-check`. Post-implementation verification: run simulator/manual smoke on at least `w390-s` and `w454-r` for the full flow `Home -> Setup -> Game -> Summary -> History -> Settings`, plus rapid-tap scoring and restart-based persistence checks.

- Success criteria: All eight target page files stop using `setTimeout`, `setInterval`, `clearTimeout`, and `clearInterval`; page-owned navigation, toast, haptic, keep-awake, and gesture interactions route through the Task `75` adapter layer; session/history/settings flows still persist correctly through the Task `76` storage path; square and round layouts continue to resolve through Task `77` screen metrics; and the main navigation flow remains stable from home through settings/history.
- Checkpoints: Pre-implementation checkpoint after Step 1 confirms the exact raw-runtime/timer inventory and whether `page/game/persistence.js` and `utils/start-new-match-flow.js` must be included; implementation checkpoint A after Step 2 confirms core pages still compile and the start/resume/setup/game/summary path passes targeted tests; implementation checkpoint B after Step 3 confirms timer grep on the migrated page files is clean and rapid-tap/manual-finish/summary-haptic behavior still passes focused validation; implementation checkpoint C after Steps 4-6 confirms history/detail/settings pages resolve router params correctly, update list data without timer assumptions, and preserve explicit confirmation behavior through page teardown and alternate taps; final checkpoint after Step 7 confirms targeted suites plus `npm run complete-check` pass and simulator/manual smoke succeeds on one square and one round family. Rollback/mitigation: if a page-level adapter swap destabilizes navigation, keep the current domain facade and limit the change to raw platform-call replacement first; if immediate persistence regresses gameplay responsiveness, keep signature dedupe and lifecycle flushes but avoid reintroducing timer-based debounce.
